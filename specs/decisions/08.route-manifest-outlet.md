# Route Manifest 与 Outlet 渲染链

## 状态

- In Progress
- 创建日期：2026-02-08
- 更新日期：2026-06-29

## 决策背景

Blueprint 早期路由主要是预览层状态：地址栏维护 `currentPath`，内部跳转只更新当前预览路径。这种做法不足以支撑真实产品：

1. 路由无法稳定映射到页面、布局、代码和资源文档。
2. Layout / Outlet / child route 只能靠局部组件约定拼出来。
3. 导航、导出、AI patch、诊断和运行时能力会各自实现一套路径理解。

当前实现已经引入 `WorkspaceRouteManifest`、`pageDocId`、`layoutDocId`、`outletNodeId` 和后端 `routeRev` 保存链路，但仍存在一个关键断层：workspace 级路由表是权威数据，画布中的 `PdxRoute` 又能独立做局部匹配。两者如果继续并行演进，会形成两套路由系统。

## 决策内容

采用 **Workspace RouteGraph** 作为项目路由的唯一权威来源。`PdxRoute` 和 `PdxOutlet` 不再被视为独立路由系统，而是 RouteGraph 的可视化投影、消费点和调试入口。

具体决策：

1. `routeManifest.root` 是 workspace 中项目级路由的唯一保存态。
2. 路由匹配、路径规范化、动态参数、fallback、导航解析、导出拓扑必须共享同一个 Route Resolver。
3. `layoutDocId` / `pageDocId` 只引用 workspace documents，不从 VFS 路径反推路由语义。
4. `PdxOutlet` 是 RouteGraph 中 layout route 的内容插槽，必须绑定到 route node 或 route region。
5. `PdxRoute` 只负责在组件内部消费当前 route context，或展示一个 route module 的局部投影；它不能成为项目级路由的第二保存态。
6. 用户操作应以 route intent 表达，例如创建页面、创建子路由、重命名 segment、移动路由、拆分 layout、绑定 outlet、删除路由。

## RouteNode 契约

当前 v1 字段：

```ts
type RouteNode = {
  id: string;
  segment?: string;
  index?: boolean;
  layoutDocId?: string;
  pageDocId?: string;
  outletNodeId?: string;
  runtime?: RouteRuntime;
  children?: RouteNode[];
};
```

`outletNodeId` 是 v1 的 default outlet 绑定。后续如果需要 named outlet，应扩展为 route-owned outlet binding，而不是让 `PdxOutlet` 自己保存任意路由字符串。

## Outlet 规则

1. 当 route node 有 `children` 且声明 `layoutDocId` 时，该 layout 文档必须提供可解析的 default outlet。
2. `outletNodeId` 若存在，必须指向 `layoutDocId` 文档内的 `PdxOutlet` 节点。
3. 若 layout 文档内有多个 Outlet，必须通过显式绑定消除歧义。
4. 缺失 Outlet 是编辑期诊断；发布和导出阶段应按严重级别阻断。
5. Outlet 注入使用 `matchChain`，而不是只用 `activeRouteNodeId + pageDocId` 的临时拼接。

## 与组件路由的关系

`PdxRoute` 可以保留为视觉组件，但它的匹配候选应来自以下两类来源：

1. 当前 workspace `matchChain` 派生出的 route context。
2. 组件 route module 的相对 RouteGraph 投影。

不再新增基于 `data-route-path` 的独立项目级路由语义。组件内需要声明路由分支时，应生成或引用 route module，而不是在任意 children 上散落路径字符串。

## 解耦原则

路由拆分方式与 VFS 拆分方式不可强绑定：

1. 一个路由可由 layout + page 多文档组合。
2. 一个文档可在不同 route node 被复用，但必须经过引用策略和诊断校验。
3. 文件路径仅用于存储与组织，不直接决定匹配语义。
4. AI、导出器、诊断器和 Inspector 都读取 RouteGraph，不扫描画布组件私有结构来推断项目路由。

## 迁移与实现方向

1. 提取共享 Route Resolver，替换编辑器、`PdxRoute`、导航 action 和导出中的临时匹配逻辑。
2. 将地址栏从“路径输入框”升级为 route tree / matchChain 操作入口。
3. 将 `PdxOutlet` inspector 的绑定动作改为 route intent。
4. 将 `PdxRoute` inspector 从编辑局部路径字符串，改为选择 route scope、route module 或调试 path。
5. 后端保存 `routeManifest` 前执行结构校验，并返回稳定 `RTE-xxxx` 诊断。

## 风险与缓解

1. 风险：`PdxRoute` 现有局部能力被误认为真实项目路由。
   - 缓解：UI 文案与 Inspector 明确区分 route module projection 与 workspace route。
2. 风险：Outlet 绑定过早复杂化。
   - 缓解：v1 只支持 default outlet，named outlet 等 route resolver 稳定后再做。
3. 风险：RouteGraph 与文档树不同步。
   - 缓解：所有 route 修改走 route intent，并在 store / backend / export 三处复用校验。

## 验收标准

- [ ] 支持 `/`、`/product/:id`、`/settings/profile` 多级路由。
- [ ] 支持 layout + child + index 组合渲染。
- [ ] `PdxOutlet` 内容来自 RouteGraph matchChain，而不是组件本地路径列表。
- [ ] `PdxRoute` 不再产生项目级第二路由源。
- [ ] 路由清单可保存、诊断、预览并导出。
